## Problem Statement
Build a private web app for personal growth that makes daily logging easy enough to sustain. The app should track three habits each day: gym, diary, and reading.

The diary entry is intentionally minimal and should be just a checkbox. Reading should capture which book was read that day, with optional notes. Gym is the main feature: workouts need to be logged from reusable templates that can stay stable for a while, then be replaced when the training plan changes.

The product should make two things feel excellent:
1. Logging today's habits, especially gym workouts, with very low friction.
2. Seeing progress clearly through a monthly calendar view.

## What Makes This Cool
This is not just another habit tracker. The useful part is the combination of:
- A smooth daily logging flow tailored to the user's real habits.
- A gym workflow that respects how training actually works, with reusable templates and quick numeric entry instead of retyping exercises every day.
- A visually strong monthly calendar that makes consistency obvious at a glance.

## Constraints
- Private, single-user app.
- Diary stays extremely lightweight: one checkbox only.
- Reading asks for the book name and optionally allows notes.
- Gym is more powerful than the other two habits.
- Gym schedules are not fixed forever, so workout templates must be replaceable over time.
- Gym templates are imported from JSON in version 1 rather than built with a full visual editor.
- Version 1 should stay balanced: complete enough to be genuinely useful, but not so ambitious that it becomes hard to finish.

## What Already Exists
- No `DESIGN.md` exists yet, so this plan defines the first UI vocabulary for the product.
- No reusable UI components or patterns exist in the repo yet.
- The only durable artifact so far is this product plan, which means the design must be explicit enough to prevent accidental UI drift during implementation.

## Premises
1. This stays a personal, private tool for one user, not a social or shared tracker.
2. The main job of the app is to make daily logging frictionless, especially for gym workouts with reusable templates.
3. The calendar is the primary surface for reviewing progress, with each date opening into that day's details.

## Engineering Decisions

### Persistence And Privacy Boundary
- Version 1 is a one-user cloud app with authentication.
- Data lives in a real backend database rather than browser-only local storage.
- Every persisted record is owned by the authenticated user.
- "Private" means account-private, not merely hidden in a single browser on a single device.
- The product should work across devices without requiring a migration from local-only storage later.

### Authentication Shape
- Version 1 uses email magic-link authentication.
- Access is restricted to a single allowlisted email address.
- The app should reject sign-in attempts from any other email even if the auth provider could otherwise create the account.
- Password management, password reset, and multi-user onboarding are out of scope for version 1.

### Application Stack
- Version 1 uses one full-stack web app rather than separate frontend and backend services.
- The app uses a relational database for persistence.
- Authentication is provided by a hosted auth service integrated into the same application flow.
- The architecture should favor boring, well-supported defaults over custom infrastructure.

### Canonical Day Model
- Version 1 uses one canonical `day` record per user per local calendar date.
- The `day` record stores the local date and the timezone context used when that date was created or edited.
- Diary, reading, and gym data attach to that canonical day record rather than deriving day boundaries on the fly from timestamps alone.
- The system should enforce uniqueness for one user + one local date so the same day cannot be created twice.

### Gym Template History Contract
- Version 1 uses per-day workout snapshots rather than a separate template revision subsystem.
- The active template defines the structure for new gym logs, but once a day is logged, that day's gym payload keeps its own embedded snapshot of the workout structure used at the time.
- Importing a new template changes future logging only; it must not rewrite past day records.
- The system may keep archived imported templates for reference, but past workout rendering must not depend on mutable template records.

### Write Semantics
- Diary, reading, and gym use module-scoped writes rather than whole-day overwrites.
- Updating one module must not clobber unchanged data in the other modules.
- Cross-device or multi-tab edits should resolve at the module boundary, not by replacing the full day payload.

### Data Ownership Shape
- The canonical `day` record stores one structured JSON payload for diary, reading, and gym subdocuments.
- Module-scoped writes patch only the relevant subdocument within that payload rather than replacing the full day record.
- The JSON shape should keep diary, reading, and gym under explicit top-level keys so boundaries remain clear in code and tests.

### Derived Status Contract
- Module completion, day strength, and calendar markers must come from one shared derived-status rule set.
- The same rules should be reused anywhere the app needs to answer "is this module complete?" or "how complete was this day?"
- Version 1 should derive these values from the canonical day payload rather than storing separate denormalized completion flags that can drift.

### Template Schema Contract
- Every imported template file must declare a schema version.
- Template files are validated server-side against a strict allowed structure before activation.
- Invalid templates must be rejected before they can become the active template for future logs.
- Extra or unknown fields should not be silently accepted in version 1.
- One shared template contract should drive validation, preview generation, and workout form rendering so field behavior cannot drift across layers.

### Test Strategy
- Version 1 uses browser-level end-to-end tests as the primary confidence layer.
- The E2E suite should cover the critical user flows end to end: sign-in, first template import, daily logging, calendar review, backfilling, and template replacement without history drift.
- Lower-level tests are secondary in version 1 and should be used sparingly for logic that would be unnecessarily brittle or opaque to verify only through the browser.
- The app must expose one deterministic test seam for current time and timezone so date-boundary scenarios can be tested reliably.
- The automated test suite must include explicit failure-path assertions for invalid template imports, failed module saves, and non-clobbering module updates.

### Calendar Query Strategy
- The month calendar should be loaded with one date-range query for the visible month rather than per-day fetches.
- Day strength and marker state should be derived from that month payload in one pass.
- Version 1 should avoid a separate precomputed calendar summary table unless real usage proves it necessary.

### Gym Save Strategy
- Gym inputs should feel instant in the UI using optimistic local state.
- Gym changes should persist to the backend in short debounced batches rather than one network write per tap.
- If a debounced save fails, the UI must show a clear unsaved/error state without discarding the user's local progress.

## Approaches Considered
### Approach A: Calendar-First Tracker
The home screen is a monthly calendar. Each day shows completion for gym, diary, and reading, and selecting a date opens the detailed daily log.

Why it works:
- Extremely easy to understand.
- Puts the calendar visualization at the center.
- Great for reviewing consistency.

Why it falls short:
- Makes today's logging slightly less direct.
- Risks making the gym template flow feel secondary.
- Does not emphasize the product's strongest differentiator: smooth structured workout entry.

### Approach B: Template-Centric Habit Journal
The app opens to today's logging flow. Gym, reading, and diary are entered from one daily screen, with gym templates as a first-class feature. A full monthly calendar remains a main tab/view for reviewing progress, backfilling days, and opening daily details.

Why it works:
- Best match for the actual use case: fast daily entry plus strong monthly review.
- Gives gym the product weight it needs without making the whole app feel like a bodybuilding dashboard.
- Keeps reading and diary simple.
- Preserves the calendar as a core feature instead of demoting it.

Why it wins:
- It combines the strongest interaction loop, today-first logging, with the strongest feedback loop, monthly progress review.
- It is the smallest complete version that already feels like the intended product.

### Approach C: Today Card + Timeline
The app opens to a compact daily check-in card with a timeline/history feed. The calendar becomes secondary.

Why it works:
- Very fast to log.
- Feels personal and journal-like.

Why it falls short:
- Weakens the importance of the month view.
- Does not match the stated desire for a clear calendar visualization.
- Introduces a more opinionated product shape with less payoff for this use case.

## Recommended Approach
Choose Approach B: Template-Centric Habit Journal.

Rationale:
- It keeps the app optimized for the real daily job: logging today quickly.
- It still gives the monthly calendar a first-class place in the product.
- It treats gym as the main feature while keeping the other habits lightweight.
- It provides a clean path for future growth, such as template versioning, personal stats, or richer training history, without forcing all of that into version 1.

## Screen Hierarchy And Navigation
Primary navigation:
- `Today`
- `Calendar`
- `Templates`

Supporting screen:
- `Day Detail` opens from `Calendar` and shows the full record for a selected date.
- `Templates` is an import-and-preview flow, not a full template-builder in version 1.

Hierarchy by screen:

### Today
1. Date header with today's completion summary.
2. Gym logging card as the dominant module on the page.
3. Reading card below gym.
4. Diary checkbox row at the bottom.
5. Secondary actions: switch gym template, edit an unfinished entry, jump to Calendar.

Why this hierarchy:
- Gym is the hardest and most valuable thing to log, so it gets the most space.
- Reading matters, but it should not compete visually with gym.
- Diary is intentionally tiny and should feel effortless, not like a journal product.

### Calendar
1. Full month grid as the dominant surface.
2. Per-day habit markers for gym, reading, and diary.
3. Selected-day preview area or panel after a date is chosen.
4. Month navigation and legend stay present but visually secondary to the grid.

Why this hierarchy:
- The job of this screen is rapid monthly scanning, not detailed editing.
- Users should understand consistency at a glance before drilling into any single day.

### Day Detail
1. Date title and completion state.
2. Gym log details first, including workout template used that day.
3. Reading details second.
4. Diary completion state third.
5. Edit and backfill actions at the end.

Why this hierarchy:
- The detail view should mirror the true weight of the habits rather than treating them equally.
- History matters for gym more than for the other two habits.

### Templates
1. Template list with one clearly marked active template.
2. JSON import area with schema guidance and validation feedback.
3. Template preview showing exercises in workout order before activation.
4. Actions to activate, replace, duplicate, or archive a template.
5. Template notes or guidance as secondary content only.

Why this hierarchy:
- The main job is managing active workout structure, not designing templates inside the UI.
- Importing a new workout plan should feel deliberate, previewable, and safe.

Navigation flow:

```text
Today (default) <----> Calendar <----> Templates
   |                     |
   | save/edit today     | tap a date
   v                     v
Today updated         Day Detail
```

Navigation rules:
- The app always opens to `Today`.
- `Calendar` is a first-class destination, not a buried secondary page.
- `Day Detail` is not top-level navigation; it is context reached from a date.
- Backfilling a day starts from `Calendar`, not from the `Today` screen.

## User Journey And Emotional Arc
Product tone:
- Motivational and energetic.
- Encouraging rather than judgmental.
- Momentum-focused rather than perfection-focused.
- The interface should feel like it wants the user to keep going today, not apologize for yesterday.

Tone rules:
- Use short, active language.
- Celebrate completion with restrained energy, not confetti or childish rewards.
- Treat missed or partial days as recoverable, not as failure.
- Make progress feel additive: "keep the week moving" is better than "you broke the streak."
- When the user returns after a gap, the interface should pull attention to today's action first and show missed days as neutral history, not a warning state.

Storyboard:

| Step | User does | User feels | Plan specifies? |
| --- | --- | --- | --- |
| 1 | Opens the app in the morning or after gym | Ready to act, should feel pulled into today's progress immediately | Yes: app opens to Today and gym is the dominant module |
| 2 | Sees today's logging screen | Oriented and slightly energized, not overwhelmed | Yes: strong hierarchy with gym first, reading second, diary third |
| 3 | Logs gym numbers from the active template | Productive and in motion | Yes: template-driven logging with low-friction numeric entry |
| 4 | Checks off diary and adds reading | Satisfied that small wins count too | Yes: lightweight logging keeps these habits easy |
| 5 | Opens the calendar | Curious about momentum across the month | Yes: full month grid is a first-class destination |
| 6 | Reviews a strong week | Rewarded by visible consistency | Yes: the calendar uses tiered day states plus fixed micro-markers for the three habits |
| 7 | Returns after a missed day or messy week | Should feel invited back, not scolded | Yes: missed days remain visible, but the UI emphasizes restarting today instead of recovering a streak |

Time-horizon design:
- First 5 seconds: the user should instantly see today's main job and feel a small surge of momentum.
- First 5 minutes: the user should be able to log a real day and review the month without confusion.
- Long-term relationship: the app should become a dependable personal ritual, not a guilt machine.

Missed-day handling:
- A missed day is shown as an unfilled date, not a failure badge.
- Recovery copy should be forward-looking, such as "Start today's log" or "Pick up where you left off."
- The app should never use loss-framed language like "streak broken" in version 1.

## Visual Direction
Visual posture:
- Athletic logbook, not soft wellness app and not corporate dashboard.
- Structured, crisp, and energetic.
- Feels like a personal training notebook upgraded into a digital tool.

Visual rules:
- Use strong typography with clear numeric emphasis so workout inputs feel serious and intentional.
- Prefer hard edges or lightly rounded shapes over bubbly cards.
- Use color for status and momentum, not decoration.
- Keep layouts tight and grid-based rather than airy and vague.
- Avoid generic SaaS patterns such as oversized hero sections, floating glass cards, or marketing-style icon grids.

UI character by screen:
- `Today` should feel active and task-oriented, with the gym module reading like a workout sheet you can complete quickly.
- `Calendar` should feel like a performance board for the month, where habit completion is legible at a glance.
- `Day Detail` should feel archival and trustworthy, like reviewing a page in a training log.
- `Templates` should feel utilitarian and structured, because the job is importing and previewing a plan, not decorating it.

Motion and feedback:
- Use short, sharp transitions for save states and screen changes.
- Completion feedback should feel satisfying but restrained, such as a color lock-in or subtle pulse.
- Avoid playful animations that make the product feel toy-like.

Calendar marker system:
- Each date cell uses a tiered background state first, then micro-markers second.
- Background state communicates the overall day at a glance:
  - Empty day: neutral background
  - Partial day: mid-emphasis background
  - Strong day: high-emphasis background
- Three fixed-position micro-markers indicate which habits were completed:
  - Gym
  - Reading
  - Diary
- The marker positions must stay consistent across every cell so the eye learns the pattern quickly.
- The cell should be readable in under a second: first "how strong was this day," then "which habits happened."

## Lightweight Design Foundation
Core design tokens:
- Color roles:
  - `Base`: neutral background for the app shell and inactive surfaces.
  - `Surface`: cards, panels, and active modules.
  - `Ink`: primary text and numeric content.
  - `Muted`: supporting labels and helper text.
  - `Momentum`: primary accent for active progress and positive completion.
  - `Warning`: import errors, validation issues, and unsynced states.
- Spacing scale:
  - Tight, compact rhythm suitable for logging tasks.
  - Use one consistent spacing ladder for all screens rather than custom one-off gaps.
- Radius:
  - Hard edges or small radius only. Avoid overly soft containers.
- Type roles:
  - Large numeric emphasis for dates, counts, and gym inputs.
  - Strong section headers.
  - Compact body text.

Component vocabulary:
- `Today` modules are stacked structured panels, not floating cards.
- Calendar cells use one consistent shape and marker layout for all states.
- Gym input rows should read like form lines in a workout sheet, not like consumer-app pills.
- Primary actions should be visually strong and singular on each screen.
- Secondary actions should stay visibly present but lower contrast.

Interaction rules:
- One dominant action per screen.
- Inline validation and save feedback over disruptive dialogs.
- Destructive actions like replacing or archiving a template should require deliberate confirmation.
- Repeated actions, such as logging several gym fields in sequence, should feel keyboard-friendly and fast.

## Responsive And Accessibility Rules
Responsive strategy:
- Mobile-first for version 1.
- Phone is the primary design target; tablet and desktop expand the same structure rather than inventing a different product.
- The default assumption is one-hand, quick-entry use.

Viewport behavior:
- On mobile:
  - Use bottom navigation for `Today`, `Calendar`, and `Templates`.
  - `Today` is a single vertical flow with gym first, reading second, and diary last.
  - `Calendar` keeps the full month grid, with day detail opening as a separate screen.
  - `Templates` stays focused on import, preview, and activation in one vertical flow.
- On tablet:
  - Preserve the same navigation model, but allow more breathing room in card width and calendar spacing.
  - `Calendar` may show a wider day preview area when space allows.
- On desktop:
  - Top or side navigation is acceptable if it stays visually secondary to content.
  - `Calendar` can use a split view with month grid on the left and selected-day preview on the right.
  - `Today` remains single-column unless extra width clearly improves gym entry speed.

Accessibility rules:
- All interactive targets should be at least 44px tall or wide on touch devices.
- Keyboard navigation must support the full product: tab order, visible focus states, checkbox toggles, date selection, and gym input entry.
- Calendar cells must be navigable by keyboard and announced clearly by screen readers with date and completion summary.
- Status must not rely on color alone; marker shape, label, or text alternative must carry meaning too.
- Form errors should be announced in plain language next to the relevant field.
- Contrast should meet accessible standards for text, controls, and status indicators.
- Motion should remain subtle and respect reduced-motion preferences.

Gym input controls on mobile:
- Default to stepper-first number input for repeated workout entry.
- Each gym row should show a numeric value with clear `-` and `+` controls plus direct typing when needed.
- The touch targets for stepper controls should be comfortably thumb-sized.
- Avoid slider-first controls for core gym logging because they are slower to adjust precisely.
- Keep input controls visually compact and aligned so logging several exercises in sequence feels rhythmic.

Gym template field model:
- Version 1 uses template-defined structured fields rather than sets-only logging or arbitrary custom inputs.
- Each exercise in the JSON template may declare a small allowed set of field types:
  - `sets`
  - `reps`
  - `weight`
  - `duration`
  - `distance`
- The app renders these field types with consistent labels, ordering, and controls across every workout.
- Templates may combine multiple allowed field types per exercise when needed.
- Version 1 should not support arbitrary freeform field schemas beyond this allowed set.

Reading model:
- Version 1 supports one book entry per day.
- The reading card contains:
  - one book title field
  - optional notes
- A day counts as reading-complete once a book title is entered.
- If the user reads more than one book in a day, version 1 assumes the primary book is enough for logging momentum.

Backfilling behavior:
- Version 1 supports backfilling, but it is a secondary workflow rather than a co-equal primary path.
- The main product posture remains same-day entry from `Today`.
- Backfilling starts from `Calendar` or `Day Detail`.
- Past-day editing should be available and reliable, but the interface should not give it the same visual weight as today's logging flow.

## Success Criteria
Version 1 is successful if:
- Logging a normal day takes less than a minute.
- Gym logging feels significantly easier than writing workout notes manually.
- The month view makes it easy to see which days included gym, diary, and reading.
- The user can swap to a new workout template without losing the history of old logged workouts.
- The app is simple enough that the user wants to return to it daily.

## NOT in Scope

- Deep analytics dashboards beyond the monthly calendar and day-level history.
